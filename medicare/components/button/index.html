    Button

    Jump to guidanceView related guidance in the CMS Design System and U.S. Web Design System

    Action buttons allow people to take actions from a current state to a future state. They are promises to the user; they must deliver the promise they offer by doing what the button says it would do.

    • PRIMARY CTA is TEAL - Use for the priority action that we want users to take. These actions are required for the user to move to the next step in the user flow (example: Enroll).
    • SECONDARY CTA is NAVY - Use for actions available but not the primary action we want users to take (example: Plan Details). There doesn't have to be a primary button on the page to use the secondary button.
    • TERTIARY CTA is OUTLINE BUTTON with the color that matches the other CTA on the page - Use for actions available but do not take priority over a primary or secondary CTAs (example: Cancel). For example: If primary button is teal, tertiary button has teal outline.
    • Outline CTAs are not used alone - They must be paired with either a primary or secondary CTA.
    • Exception: CTA over background hero image on landing pages.
    • Lower priority actions are links - Refer to link styling guidelines under Typography.
    • Landing Page Button - This larger button is ONLY used in the hero image of landing pages.

    View more guidance and design assets at Medicare.gov DSM.

    Button classes

    • Mobile buttons - Consider using full-width buttons on mobile devices. This will help improve the click range for both left or right-handed users. Use the class mds-c-button--mobile-full-width to apply full-width for mobile and the responsive margin classes to adjust margins on mobile.
    • Hiding buttons on print - Hide buttons when printing to avoid unnecessary content for printed material. Use the ds-u-visibility--hidden-print class to hide buttons or any other elements on print. Refer to the display/visibility utilities for more information.

    Button icons

    • Add an inline SVG icon and it will become the same color as the button text. For the crispest icon rendering, ensure the icon has a square viewBox with values that are multiples of 8 (ie. 24x24).
    • Use the margin utility class to add spacing between the icon and button text.

    The Button component accepts its text as children (AKA inner HTML), which means you can also pass in HTML or custom components. This gives you a lot of flexibility and supports a variety of advanced use cases. The most common use case would be passing in an SVG icon along with the text.

    In addition to the supported props listed, you can also pass in additional props, which will be passed to the rendered root component. For example, you could pass in a target prop to pass to the rendered anchor element.

    Guidance

    Design Guidelines

    • Action buttons do NOT include icons.
    • Exception: Can include icons in buttons in mobile for CTAs outlining functionality. Examples: Filter, Sort, Map, and Menu.
    • Other exceptions require advance and separate approval than mock-ups by product teams. See "Exception process" in Foundations.
    • Buttons use title case. Example: Join Plan or Find Doctors.
    • Note that clickable cards are sentence case. See that component for more information.

    When pairing 2 CTAs:

    • In forms/decision trees: Have the preferred action be on the right. Example: Back/Next or Cancel/Submit.
    • All other cases: Have the preferred action be on the left. Example: Create Account/Return to Home or Change My Account/Stop Easy Pay.

    Content Guidelines

    • Describe what will happen, not the current state. Example: Start Over.
    • CTA buttons always start with an action verb that describes the main thing users will be able to do once selected.
    • Pair it with a noun to help it be more clear. Example: Change Password or Add Drug.
    • If in a form, it can be just the action verb if it's navigating between steps. Example: Cancel, Back, Next, Submit.
    • Use the least amount of words possible (no more than 4).
    • Use title case for CTA buttons.

    Get more content guidelines by going to the buttons page in the Medicare.gov DSM.

    Accessibility

    • Buttons should display a visible focus state when users tab to them.
    • Create a button with a button or a element to retain the native click functionality. Avoid using <div> or <img> tags to create buttons. Screen readers don't automatically know either is a usable button.
    • When styling links to look like buttons, remember that screen readers handle links slightly differently than they do buttons. Pressing the Space key triggers a button, but pressing the Enter key triggers a link.
    • Dimmed or unavailable buttons should have the disabled attribute applied. This removes native click and keypress events from the button. It also prevents automated scanners from logging a low contrast error. Finally, it announces the button as "dimmed" or "disabled" to screen readers, offering users additional context.
    • Disabled buttons - Don’t disable buttons, unless there’s a good reason to.
    • If you do disable a button, make sure it receives the disabled styling. A button element will automatically be styled as a disabled button when it has the disabled HTML attribute, but an a element will need to have the .ds-c-button--disabled class applied to it.
